# v2.3.1 - Hotfix: runner crashloop on missing python-multipart + safe_update rebuild gap

Date: 2026-04-29

## What was broken in v2.3.0

A real prod-blocking pair of bugs surfaced the moment an admin clicked
"Backup erstellen" on the new `/admin/backups` page:

1. **The `update_runner` container couldn't start** because
   `apps/update_runner/requirements.txt` forgot to declare
   `python-multipart`. FastAPI's `/backups/upload` route uses `Form()` +
   `UploadFile`, so loading the app raised `RuntimeError: Form data
   requires python-multipart to be installed` at import time and uvicorn
   exited with non-zero. Tests passed locally only because the runner
   piggybacked on the api's venv (which has multipart for thread-message
   uploads).

2. **`safe_update.sh` never rebuilt the runner image.** Every release
   prior to v2.3.0 left the runner unchanged, so the script's hard-coded
   `docker compose build api` + `docker compose up -d --build api
   api_worker web` worked fine. v2.3.0 was the first release to add new
   code to the runner, but the rollout silently used the cached
   pre-v2.3.0 runner image — the api was on v2.3.0 and tried `POST
   /jobs/backup`, the stale runner answered 405 on its old `/jobs/{id}`
   GET-only route, and the admin UI surfaced "Runner returned HTTP 405:
   Method Not Allowed".

The combination meant: even a manual `docker compose up -d --build
update_runner` after v2.3.0 would have flipped from "wrong code" to
"crashloop", hitting bug 1.

## What v2.3.1 changes

### 1. `apps/update_runner/requirements.txt` (`fix(runner)`)

Adds `python-multipart==0.0.20`. Comment in the file explains why it
can't be omitted so a future "let me trim deps" PR doesn't reintroduce
the regression.

### 2. `scripts/safe_update.sh` (`fix(ops)`)

`docker compose build api` becomes `docker compose build api
update_runner`, and `docker compose up -d --build api api_worker web`
becomes `... api api_worker web update_runner`. The `wait_for_service_health`
loop also now waits on `update_runner`. Comments in the file explain
why the runner has to be in both lists.

### 3. New regression test (`test(runner)`)

`apps/update_runner/tests/test_main.py::test_runner_app_exposes_upload_endpoint`
asserts that `/backups/upload` is registered on the FastAPI app. If
`python-multipart` is ever missing from the runner's deps again,
FastAPI silently skips the route during import — this test would catch
that immediately rather than waiting for a prod admin to click the
button.

## Compatibility

None broken. v2.3.1 is a strict superset of v2.3.0:
- Existing `/admin/backups/database` (DB-only download) unchanged.
- Existing `/admin/backups/*` routes from v2.3.0 unchanged in shape; the
  fix just makes them actually work end-to-end on prod.
- No DB migrations.
- No new env vars beyond v2.3.0's `BACKUP_PASSPHRASE`.

## Rollout

Standard `safe_update.sh --pull --branch main`. Because v2.3.1 also
patches `safe_update.sh` itself, the FIRST run after the pull will use
the OLD safe_update.sh script (which doesn't rebuild update_runner) —
but that's fine because `git pull` already replaces the script before
it runs. Any subsequent runs use the new version. To verify after the
upgrade, the runner container's `Created` timestamp should be newer
than the api's `Created` timestamp.
