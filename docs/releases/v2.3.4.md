# v2.3.4 - Audit retention, CSV export, brute-force alerts, cleaner backup warning count

Date: 2026-05-01

## What's new

Four operationally-focused additions that don't change the user-facing
app surface but make the system easier to live with at the admin layer:

1. **Cleaner backup warning count** — the success card now shows
   "0 warnings" for the common benign case (tar reporting concurrent
   writes to the upload volume) and reserves the count for messages
   that genuinely warrant attention. No more "1 warning" on every
   backup just because the api_worker happened to be downloading a
   material image at the same time.

2. **Audit log retention policy** — a new daily worker job prunes
   `audit_logs` rows older than the configured threshold. Default is
   730 days (~ 2 years) so existing rows aren't touched on upgrade;
   set to 0 to disable. Runs at 03:00 local time; idempotent across
   worker restarts via an AppSetting bookmark.

3. **Audit log CSV export** — admins can now download the audit log
   as CSV from the Audit tab, with the same filters as the in-page
   view (category multi-select + date range). Useful for compliance
   reports, off-line analysis, and email-handing-to-auditors. No row
   cap.

4. **Brute-force login alerts** — when enabled, the api raises a
   Telegram and/or email alert when repeated `auth.login_failed`
   events cross a threshold (default 5 failures per email within 5
   minutes, OR 10 failures from the same IP within 60 seconds).
   Audit-log-anchored dedup prevents double-firing across worker
   processes or restarts. Disabled by default — opt in via the
   `AUDIT_ALERTS_*` env vars.

## Detailed changes

### 1. `scripts/backup.sh` — `fix(backups)`

Inverts the warning-count regex from "match this benign pattern" to
"match anything BUT this benign pattern". Specifically excludes:

```
file changed as we read it
File removed before we read it
```

Both are normal artifacts of tar archiving a directory with
concurrent writers. The file is still in the archive whole, just
from a moment-earlier snapshot. If a TRULY anomalous tar message
appears (permission denied, disk full, bad bytes), it now gets
counted, surfaces in the success card, and prompts a real "look at
the log tail" investigation.

### 2. Audit retention service — `feat(audit)`

New `apps/api/app/services/audit_retention.py`:
- `prune_audit_logs(db, retention_days=N)` — core helper, deletes rows
  older than the threshold, returns count + cutoff.
- `prune_audit_logs_if_due(db)` — worker entry point with feature-flag
  + once-per-local-day gating + crash-resistant bookmark.
- `count_purgeable(db, retention_days=N)` — for the admin UI's
  "next prune will delete N rows" preview (not yet wired into UI;
  endpoint can be added when needed).

`apps/api/app/worker.py` calls the if_due variant on the existing
1-minute poll cadence. Settings:
- `audit_log_retention_days` (default 730)
- `audit_log_retention_run_hour_local` (default 3)

### 3. Audit CSV export — `feat(audit)`

New `apps/api/app/routers/admin.py::export_audit_logs_csv`:
- `GET /api/admin/audit-logs/export.csv?from=&to=&category=`
- Same filter shape as the in-page view (`from`/`to` ISO datetimes,
  `category` repeatable). No row cap on the export.
- Streaming response with proper `Content-Disposition: attachment`
  + `Cache-Control: no-store` + cleanly-named filename
  (`audit-logs-YYYYMMDD-HHMMSS.csv`).
- Permission gate: existing `audit:view`.

Frontend (`apps/web/src/App.tsx` + `pages/AdminPage.tsx`):
- New `exportAuditLogsCsv(options)` action in `App.tsx` that fetches
  with the auth bearer header, reads the response as a blob, and
  triggers a browser save-as via a synthetic anchor.
- New "CSV exportieren" / "Export CSV" button next to "Refresh" in
  the Audit tab toolbar. Translates the active in-page filters into
  the ISO bounds + category list the endpoint expects.

### 4. Brute-force login alerts — `feat(auth)`

New `apps/api/app/services/auth_alerts.py`:
- `evaluate_after_failed_login(db, email, ip)` — runs after each
  `auth.login_failed` audit row commits.
- Detection rules:
  - Rule A: ≥ N failures for the same email within
    `audit_alerts_failures_per_email_window_seconds` (default 5 / 300s)
  - Rule B: ≥ M failures from the same IP within a 60-second window
    (default 10)
- Audit-log-anchored dedup: writes a `auth.alert_brute_force` row
  before sending; subsequent failures within
  `audit_alerts_dedup_window_seconds` (default 600s) skip dispatch.
  Works correctly across multiple api workers since the dedup state
  is in the DB.
- Channels: existing `services/telegram.py` + `services/emailer.py`.
  Each toggleable independently.

`apps/api/app/routers/auth.py::login` calls the evaluator inside a
try/except so a bug in the alert path can never block a legitimate
login response with a 500.

### 5. New settings (all default-off)

Added to `apps/api/app/core/config.py`:

```
audit_log_retention_days                       = 730
audit_log_retention_run_hour_local             = 3

audit_alerts_enabled                           = False
audit_alerts_failures_per_email_threshold      = 5
audit_alerts_failures_per_email_window_seconds = 300
audit_alerts_failures_per_ip_threshold         = 10
audit_alerts_dedup_window_seconds              = 600
audit_alerts_send_telegram                     = True
audit_alerts_send_email                        = False
audit_alerts_email_recipient                   = ""
```

## Migrations

None. All changes are additive (new settings, new endpoints, new
service modules). The existing `audit_logs` table is reused; no
schema changes.

## Compatibility

- All four features are off-by-default or use safe defaults that
  match prior behavior. Existing deployments see zero behavior change
  on upgrade unless the new env vars are explicitly set.
- The CSV export uses the existing `audit:view` permission — no new
  permissions to grant.
- `audit_log_retention_days = 730` matches "we keep two years" — most
  audit rows on existing prod are well under this threshold so the
  first prune cycle on a normal-sized DB will likely be a no-op.

## Rollout

Standard `safe_update.sh --pull --branch main`. The v2.3.3 fixes
(`stdbuf` + `--force-recreate update_runner`) are in place so the
runner image will rebuild cleanly with the new `scripts/backup.sh`
warning filter. After upgrade, optionally enable brute-force alerts
in your prod env:

```
AUDIT_ALERTS_ENABLED=true
AUDIT_ALERTS_SEND_TELEGRAM=true
# (Telegram bot already configured for other features works here too)
```

Then `docker compose up -d api api_worker` to pick up the new env.
