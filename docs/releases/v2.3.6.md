# v2.3.6 - Hotfix: backup.sh exited 1 when all tar output was benign noise

Date: 2026-05-01

## What was broken in v2.3.4+

`scripts/backup.sh` aborted with `exit_code=1` immediately after the
`Upload-Archiv` stage in the common case where the only tar stderr
output was `file changed as we read it` from concurrent
material-image fetches. The encrypted archive was actually written and
valid — but the script bailed before emitting the `::SMPL_SUMMARY:`
marker, so the runner reported the job as failed and the in-UI
backup/restore flow refused to proceed.

This bug has been latent in v2.3.4 and v2.3.5 since the warning-count
logic was changed to "count messages that are NOT in the benign list".
It only manifests when the script is run via the runner (which has
`bash -c`-style stricter quoting + freshly-recreated containers) AND
the api_worker happens to be writing to `/data/uploads/material_catalog_images/`
during the tar — both of which were dormant during my v2.3.4/v2.3.5 host-shell
rollouts but bit immediately on the in-UI updater test.

## Root cause

The new warning-count pipeline:

```bash
WARNINGS=$(
  grep -v -E "$TAR_NOISE_PATTERN" "$TAR_STDERR" 2>/dev/null \
    | grep -v -E '^[[:space:]]*$' \
    | wc -l
)
```

`grep -v PATTERN` exits **1** when EVERY input line matched (i.e., all
input was the benign pattern). With `set -euo pipefail` active at the
top of `backup.sh`, that exit-1 inside the pipeline:

  1. Made the pipeline as a whole fail (pipefail).
  2. Made the command-substitution `$(...)` propagate the exit code.
  3. Triggered `set -e`, which killed the entire script.

So the most common case the v2.3.4 logic was designed to handle —
"all output is benign, count should be 0" — was the exact case that
killed the script.

## Fix

Wrap each `grep -v` so its all-matched exit-1 doesn't poison the
pipeline:

```bash
WARNINGS=$(
  { grep -v -E "$TAR_NOISE_PATTERN" "$TAR_STDERR" 2>/dev/null || true; } \
    | { grep -v -E '^[[:space:]]*$' || true; } \
    | wc -l
)
```

The braces preserve pipeline order; the `|| true` per-grep fallback
turns "all matched" from a fatal exit into a no-op pass-through with
zero output. `wc -l` still computes the right count; the WARNINGS
variable still holds 0 in the all-benign case as intended.

Verified against three test cases:
- Empty stderr (no tar output): WARNINGS=0 ✓
- Only-benign stderr (one or more `file changed` lines): WARNINGS=0 ✓
- Real anomaly mixed in (e.g. `Permission denied`): WARNINGS=N ≥ 1 ✓

All pipefail-safe.

## Compatibility

- One-line fix in `scripts/backup.sh`. No new settings, no new
  permissions, no DB migrations.
- The fix is invisible on a backup that has truly anomalous tar
  output — those still get counted and surface in the UI.
- All four v2.3.4 features (audit retention / CSV export / brute-force
  alerts / cleaner warning count) are still active. Only the bash
  bug got fixed; the design is unchanged.

## Rollout

Standard `safe_update.sh --pull --branch main` from the host shell.
The in-UI updater path will work for the first time after this lands —
v2.3.5 fixed the `safe.directory` issue, v2.3.6 fixes the backup-step
abort. The next runner-mediated update should complete end-to-end with
visible progress markers.
