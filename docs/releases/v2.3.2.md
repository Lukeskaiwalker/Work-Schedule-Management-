# v2.3.2 - Real backup progress tracking

Date: 2026-05-01

## What this fixes

The v2.3.0 admin Backups page rendered an animated-but-fake progress bar
during a backup run and a bare green checkmark on success. An operator
clicking "Backup erstellen" had no way to tell whether the job was
actually doing something or about to fail — and on success they had no
idea how big the backup was, how long it took, or whether anything was
flagged.

This release adds **real, measured progress tracking** to the backup
flow plus a structured success summary card. Restore jobs benefit from
the same plumbing automatically (they share the runner's job model);
restore-side stage markers can be added to `scripts/restore.sh` later
without further code changes elsewhere.

## What's new

### 1. `::SMPL_STAGE:` / `::SMPL_SUMMARY:` marker protocol (`feat(scripts)`)

`scripts/backup.sh` now emits structured progress lines at every phase:

```
::SMPL_STAGE: ensure_containers 5 Container vorbereiten
::SMPL_STAGE: db_dump 25 Datenbank-Dump
::SMPL_STAGE: archive_volume 60 Upload-Archiv
::SMPL_STAGE: manifest 80 Manifest schreiben
::SMPL_STAGE: encrypt 90 Verschlüsselung
::SMPL_STAGE: done 100 Fertig
::SMPL_SUMMARY: filename=backup-20260501-145012.tar.enc \
                size_bytes=3671152672 \
                duration_seconds=492 \
                warnings=1
```

Markers are deliberately simple and human-readable so they show up in
the log tail alongside the rest of the output as breadcrumbs. Format
is documented inline in `scripts/backup.sh` so other scripts can adopt
the same protocol when they want progress signaling.

### 2. Runner parses markers in real time (`feat(runner)`)

`apps/update_runner/app/jobs.py`:
- `Job` gains `stage`, `stage_label`, `progress_percent`, plus four
  `summary_*` fields populated from the SUMMARY marker.
- `_run_subprocess` switched from `subprocess.run(stdout=log)` to
  `subprocess.Popen(stdout=PIPE)` + line-by-line iteration so it can
  observe each stdout line as it appears, parse markers via
  `_maybe_parse_marker`, and still write the original line to the log
  file unchanged.
- Marker parser is intentionally forgiving: malformed percent values
  fall through (previous progress persists), unknown summary keys are
  ignored, and lines that don't match the prefix are pure pass-through.

`apps/update_runner/app/main.py`'s `JobResponse` now exposes the seven
new fields. Older clients ignore them.

### 3. UI surfaces the progress (`feat(web)`)

`apps/web/src/components/admin/AdminBackupsPanel.tsx`'s `BackupJobBanner`:
- **While running**: shows the stage label ("Datenbank-Dump", etc.) +
  a measured percent + a real progress bar that fills as the script
  reaches each marker. The bar gracefully falls back to an animated
  indeterminate stripe for older runners that don't emit markers.
- **On success**: replaces the bare checkmark with a green summary card
  showing the filename, file size (e.g. "3.42 GB"), duration
  ("8m 12s"), and warning count ("1 warning") if any.
- **On failure**: shows the runner's `detail` message in a red banner
  AND auto-expands the log tail so the operator can diagnose without
  clicking. The log tail is collapsed-by-default during a successful
  run to keep the panel clean.

### 4. Backwards compatibility

- Pre-v2.3.2 clients (older browsers caching v2.3.0/v2.3.1 frontend)
  ignore the new fields and continue to render the old animated bar.
- The runner is fully forward-compatible: any script that doesn't emit
  markers (e.g. the existing `scripts/restore.sh`) just produces a
  job with `stage=null progress_percent=null`, and the UI shows the
  indeterminate bar — same UX as before this release.
- No new permissions, no new endpoints, no DB migration.

Total runner suite: 36/36 passing.

## Rollout

Standard `safe_update.sh --pull --branch main`. The v2.3.1 fix to
`safe_update.sh` (rebuild update_runner alongside api/web) is now
in place on prod, so the runner image WILL be rebuilt this time and
the new marker parser will land cleanly in the running container —
no more bootstrap dance.

After the rollout, click "Backup erstellen" on `/admin/backups` to
see the new progress bar in action. A 3 GB backup typically takes
6–10 minutes; you should see the bar move through:

  5% Container vorbereiten
  → 25% Datenbank-Dump
  → 60% Upload-Archiv
  → 80% Manifest schreiben
  → 90% Verschlüsselung
  → 100% Fertig

with the success card showing size + duration + warning count when done.

## Deferred to v2.3.3

The audit-log enhancements discussed alongside this release —
retention policy, CSV export, brute-force alert webhook — were
intentionally split out so this release stays focused on the active
UX gap (backup status visibility). v2.3.3 will land those next.
