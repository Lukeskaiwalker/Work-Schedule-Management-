# v1.7.3 - Mixed-key attachment recovery

Date: 2026-03-17

## Highlights

- Added read-only fallback support for legacy file-encryption keys.
- New uploads continue to use the current `FILE_ENCRYPTION_KEY`.
- Older attachments encrypted under a previous key can now be previewed and downloaded again when that old key is provided through `FILE_ENCRYPTION_LEGACY_KEYS`.

## Root Cause

- The current repo and current server both use the same active `FILE_ENCRYPTION_KEY`.
- The server still had an older `apps/api/.env` file with a different encryption key.
- That older key successfully decrypts the broken uploads created from 2026-02-22 through 2026-03-09, while the current key decrypts uploads created from 2026-03-10 onward.
- The app had been switched to `apps/api/.env.example`, so the previous key was no longer available at runtime for file reads.

## Upgrade Notes

- No new Alembic migrations are introduced.
- Configure any recovered historical keys as a comma-separated list:
  - `FILE_ENCRYPTION_LEGACY_KEYS=<old-key-1>,<old-key-2>`
- Standard rollout:
  - `./scripts/update_release_metadata.sh`
  - `docker compose up -d --build api api_worker web caddy`

## Validation Baseline

- `./scripts/test.sh` passes (`83 passed` backend tests + web production build pass).
