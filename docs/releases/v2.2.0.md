# v2.2.0 - Calendar absences, customer fields, chat reactions, daily clock summary

Date: 2026-04-27

## Highlights

- **Personal time-tracking calendar now shows absences.** Approved Urlaub, Krankheit, Berufsschule, Feiertag and other absence types render as color-coded pills on the day cells of the user's calendar — they used to silently disappear. Recurring entries (weekly Berufsschule etc.) materialise on each matching weekday.
- **Customer card learned two new fields.** Optional **Geburtstag** for individual contacts and **Marktakteur-Nummer** (Marktstammdatenregister) for PV / energy customers. Both surface in the Kontaktdaten card and the customer create/edit modal; opt-in per customer with no PV-categorisation enforced yet.
- **Office Aufgaben view has a task name search.** Free-text search at the start of the filter row finds tasks by title or description, client-side over the already-fetched list — no backend round-trip per keystroke.
- **Threads now support multiple pictures per message.** Pick several files at once; each shows as a removable chip in the composer and a separate attachment on the bubble.
- **In-thread image lightbox.** Clicking any picture opens a WhatsApp-style full-viewport viewer with ESC-to-close, ←/→ to step through siblings of the same message, and a mobile-tuned layout.
- **Emoji reactions on thread messages.** Click an existing bucket to toggle, or "+" to pick from a palette of common emojis. Buckets show the count and highlight when you reacted.
- **Daily clocked-in summary.** End-of-day digest of who is still clocked in plus today's worked hours per user, dispatched via Telegram and/or email at a configurable local time. Useful for spotting people who forgot to clock out.

## Detailed changes

### 1. Calendar absence display fix (`fix(time)`)

`apps/web/src/pages/TimePage.tsx`'s `monthCalendar` memo now merges
`approvedVacationRequests` and `schoolAbsences` into the day cells,
scoped to the active target user (self by default; the manager's
selected target when applicable). The Python→JS weekday conversion
mirrors the backend's `_expand_school_absence_days` rule, so recurring
school absences materialise on the same days the planning endpoint
would surface server-side.

A new `AbsenceMarker` is attached to each cell; the renderer shows a
compact pill with an `aria-label` summary and tints the cell
background by absence type. Existing hours display is preserved when
both an absence and worked hours are present.

### 2. Customer birthday + Marktakteur-Nummer (`feat(customer)`)

Two optional columns on the `customers` table:

- `birthday: Date | null`
- `marktakteur_nummer: String(64) | null`

Migration `20260605_0050_customer_birthday_marktakteur.py` adds both
nullable columns; no backfill needed. Pydantic Create/Update/Out
schemas, the `CustomerWriteInput` client type, the create/list-item
helpers in `workflow_customers.py`, the `CustomerContactCard` rows,
and the `CustomerModal` form are all extended consistently.

The frontend's birthday input is `<input type="date">` so it
round-trips ISO `YYYY-MM-DD` directly into Pydantic's `date`. The
contact card formats it via `toLocaleDateString` for display.

`marktakteur_nummer` is stored as String(64) — wider than the actual
MaStR ID format (`SEE901234567890`, ~15 chars) so future format
changes don't trip a schema migration. Hint text explains the field
in the tooltip.

### 3. Office task name search (`feat(office)`)

Local state in `OfficeTasksPage.tsx`; no context plumbing. The
existing `visibleTasks` memo gains a case-insensitive substring
filter against `title` (primary) and `description` (fallback) so
users can search by either headline or a remembered detail. The
"Filter zurücksetzen" button clears the new field alongside the
others.

### 4. Multi-picture thread upload (`feat(chat)`)

The composer's file input is now `multiple`. State moved from
`messageAttachment: File | null` to `messageAttachments: File[]`,
with append-on-pick + dedup so iterative selections build up the
right list. Each picked file shows as a removable chip; a
"Clear all" link appears when more than one is queued.

The `POST /api/threads/{id}/messages` endpoint now accepts
`attachments: list[UploadFile]` (repeating multipart field) **and**
keeps the legacy single `image` / `attachment` fields for older
clients — both code paths fold into the same loop, producing N
Attachment rows in selection order.

### 5. In-thread image lightbox (`feat(chat)`)

New `components/chat/MessageImageLightbox.tsx` component. Clicking
any image attachment opens a full-viewport overlay; ESC and backdrop
click close it; ←/→ step through other images in the *same* message
bubble for a small-gallery feel. Mobile-tuned: arrow buttons reflow
inside the stage at narrow widths; a caption shows the file name and
"i / N" position.

The image attachment in the message bubble is now a `<button>`
wrapping the `<img>` so it's keyboard-focusable and announceable as
"open image preview".

### 6. Emoji reactions on messages (`feat(chat)`)

New `message_reactions` table with a UNIQUE `(message_id, user_id,
emoji)` constraint — one user can react with multiple distinct emojis
to the same message but cannot double-stack. Cascades from `messages`
and `users` so deletions tidy up.

Endpoints:

- `POST /api/messages/{id}/reactions {emoji}` — adds the current
  user's reaction (idempotent).
- `DELETE /api/messages/{id}/reactions?emoji=X` — removes it.

Both return the refreshed `MessageOut`, which now carries an
aggregated `reactions` list (one bucket per emoji with `count`,
`user_ids`, `me_reacted`) so the frontend renders "👍 3 (you reacted)"
without grouping client-side.

The `MessagesPage` bubble shows a reaction strip below the
attachments. Each bucket toggles the current user's reaction;
a "+" opens a small palette (👍 ❤️ 😂 🎉 👏 🔥 🙏 👀) to add a new one.
Tinted state indicates "you reacted" so users can re-click to retract.

Migration: `20260605_0051_message_reactions.py`.

### 7. Daily clocked-in summary (`feat(time)`)

New service `app/services/daily_clock_summary.py`:

- `build_clock_summary_for_date(db, target)` returns the structured
  summary (clocked-in users with their start time and worked-hours-
  so-far, plus totals). Hours-today is computed live, clamping open
  entries to `now()` so values match the on-screen counter.
- `format_clock_summary_text(summary, language)` renders the
  Telegram/email body with a 📋 header, the still-clocked-in list,
  and totals.
- `dispatch_daily_clock_summary(db, target_local_date)` ships the
  summary via the configured channels.
- `dispatch_daily_clock_summary_if_due(db)` is the worker entry point
  with feature-flag, time-of-day, and once-per-day-bookmark gating.

The bookmark advances *before* external sends so a crash means at
most one missed digest, never duplicate spam.

`app/worker.py` calls `dispatch_daily_clock_summary_if_due` once per
minute on its existing poll loop. New settings:

- `daily_clock_summary_enabled` (default `false`)
- `daily_clock_summary_target_hour_local` / `_target_minute_local`
  (default 18:00)
- `daily_clock_summary_send_telegram` (default `true`)
- `daily_clock_summary_send_email` (default `false`)
- `daily_clock_summary_email_recipient`

On-demand admin endpoints:

- `GET  /api/time/clock-summary[?target_date=YYYY-MM-DD]` — JSON
  summary for the "wer ist noch eingestempelt?" admin panel
  (gate: `time:view_all` or `time:manage`).
- `POST /api/time/clock-summary/dispatch[?target_date=YYYY-MM-DD]` —
  send right now via the configured channels (gate: `time:manage`),
  returning per-channel success flags so the UI can render
  "Telegram: ✓ / Email: ✗ (SMTP not configured)".

## Migrations

Two additive migrations land in this release:

- `20260605_0050_customer_birthday_marktakteur` — adds
  `customers.birthday` (Date, nullable) and
  `customers.marktakteur_nummer` (String(64), nullable). No backfill.
- `20260605_0051_message_reactions` — creates the
  `message_reactions` table with the UNIQUE
  `(message_id, user_id, emoji)` constraint and ON DELETE CASCADE
  FKs. No backfill.

Both are safe to apply on a populated production DB.

## Compatibility / breaking changes

None expected.

- The chat upload endpoint keeps the legacy `image` / `attachment`
  single-file form fields working alongside the new `attachments`
  list, so any external automation continues to function.
- All new customer fields are nullable; existing rows pass through
  unchanged.
- All new chat features are off-by-default for stale clients (older
  frontends just won't show the reaction strip / lightbox, and never
  call the new endpoints).
- The daily summary feature is `daily_clock_summary_enabled = false`
  by default — no automatic Telegram messages start firing on
  upgrade until an admin opts in.

## Rollout

Standard `safe_update.sh --pull --branch main` from prod. The new
`update_release_metadata` build-arg fix from v2.1.0's defensive patch
(commit `9f278c4`) means the api image has accurate
`/app/.release.env` baked in regardless of host file presence.

After upgrade, optionally configure the daily summary feature in
production by setting:

```
DAILY_CLOCK_SUMMARY_ENABLED=true
DAILY_CLOCK_SUMMARY_SEND_TELEGRAM=true
DAILY_CLOCK_SUMMARY_TARGET_HOUR_LOCAL=18
```

Telegram requires `TELEGRAM_BOT_TOKEN` + `TELEGRAM_CHAT_ID` (already
present in the existing config). Email requires SMTP settings (also
already present) plus `DAILY_CLOCK_SUMMARY_SEND_EMAIL=true` and
`DAILY_CLOCK_SUMMARY_EMAIL_RECIPIENT=<email>`.
