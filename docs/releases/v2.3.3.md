# v2.3.3 - Hotfix: backup progress markers now actually visible during the run

Date: 2026-05-01

## What was broken in v2.3.2

The new ::SMPL_STAGE: progress markers reached the runner only at the
very end of a backup run — never during the multi-minute middle phases.
Operators saw the panel sit on "Vorbereitung… (animated bar)" for ~8
minutes, then suddenly jump to "Fertig" + the success summary card.
The intended UX (live "Datenbank-Dump… 25%" → "Upload-Archiv… 60%" →
"Verschlüsselung… 90%") was effectively invisible.

## Root cause

When bash's `printf` writes to stdout that's been piped to a parent
process (the runner reads via `subprocess.Popen(stdout=PIPE)`), libc
defaults to **full block buffering** (4 KB) instead of line buffering.
All the marker lines accumulated in bash's stdout buffer. The buffer
only flushed when:

1. It filled up (4 KB ≈ ~80 marker lines, never reached on a normal run), OR
2. The script exited and libc auto-flushed on shutdown.

So in practice, every marker came through in a final burst right before
the script exited — making the runner's `Job.stage` field flip from
`null` straight to `done` with no intermediate state.

This is a classic gotcha when bridging shell scripts and parent processes
via pipes. It only shows up under specific conditions (output to a pipe,
not a terminal; printf using libc-buffered FILE\*; long-running script
between markers) — none of which the unit tests caught because the test
stubs synthesize stdout content all at once.

## What v2.3.3 changes

### 1. `apps/update_runner/Dockerfile` (`fix(runner)`)

Adds `coreutils` to the Alpine package list. The Alpine base image
ships BusyBox, which doesn't include the GNU `stdbuf` helper. With
`coreutils` installed, `stdbuf` is on PATH and we can use it as the
buffer-mode override below.

This adds ~3 MB to the image (Alpine `coreutils` 9.4-r4) — acceptable
trade-off for correct progress signaling.

### 2. `apps/update_runner/app/jobs.py` (`fix(runner)`)

`_run_subprocess` now prepends `["stdbuf", "-oL", ...]` to every
subprocess command. `stdbuf -oL` is GNU coreutils' wrapper that uses
LD_PRELOAD to override libc's stdout buffer mode in the child process
to **line-buffered** — meaning libc flushes after every `\n`. Now the
runner sees each `::SMPL_STAGE:` marker the moment bash emits it,
which propagates through the api and into the UI within one polling
interval (≤2 s).

Applied to all three subprocess kinds (update / backup / restore) so
when `scripts/restore.sh` adopts the marker protocol later, it gets
correct buffering for free.

### 3. `scripts/safe_update.sh` (`fix(ops)`)

Splits the single `docker compose up -d --build api api_worker web
update_runner` line into two:

```bash
docker compose up -d --build api api_worker web
docker compose up -d --build --force-recreate update_runner
```

Why specifically `update_runner` gets `--force-recreate`: Docker
Compose's image-change detection sometimes misses when only the image
content changes but the tag stays the same (`smpl-all-update_runner:
latest` → `smpl-all-update_runner:latest` — same tag, different SHA).
We hit this on the v2.3.1→v2.3.2 prod rollout: the new image was
built but the running container kept its older image, leaving the
v2.3.2 marker parser absent until a manual force-recreate. Forcing
recreation on the runner specifically avoids this trap without
churning api/api_worker/web unnecessarily on every release.

### 4. Regression test

`apps/update_runner/tests/test_main.py::test_backup_job_queued_writes_log`
gains an assertion that the captured subprocess command starts with
`["stdbuf", "-oL"]`. If a future refactor strips the wrapping,
the test fails loudly at CI rather than waiting for an operator to
notice missing progress mid-rollout.

## Compatibility

- No new env vars, no new permissions, no DB migrations.
- The `stdbuf` wrapping is invisible to scripts: from inside the script
  it looks identical to the previous behavior.
- Pre-v2.3.3 frontends (cached browsers) continue to work — the
  buffering fix is server-side only.
- The Dockerfile change to add `coreutils` rebuilds the runner image
  fresh; older runner images without coreutils will fail to start
  v2.3.3+ code (the runner unconditionally prepends `stdbuf`). This
  makes the rollout dependency explicit: you must rebuild the runner
  image for v2.3.3 to work.

## Rollout

Standard `safe_update.sh --pull --branch main`. The v2.3.2 version of
`safe_update.sh` (already on prod) does include `update_runner` in
its `docker compose build` step — so the runner image WILL get
rebuilt, and Compose's image-change detection now triggers a recreate
because the tag content changes (the new image has the coreutils
package).

To verify after upgrade, click "Backup erstellen" on /admin/backups
and watch the stage label tick through:

  Container vorbereiten → Datenbank-Dump → Upload-Archiv →
  Manifest schreiben → Verschlüsselung → Fertig

with the bar filling smoothly between stages, not flat-lining at 0%
until the very end.
