# v2.0.0 - Customers, Partners, Werkstatt, and Project Gantt

Date: 2026-04-22

## Highlights

- **Customer master data** is now a first-class entity. A new **Kunden** sidebar entry manages all customers in one place; projects link to them via a searchable combobox in the project modal with inline create. Customers with multiple open projects are no longer re-keyed for each one.
- **External partners** (electricians, plumbers, roofers and other subcontractors) can now be assigned to tasks alongside internal employees. A new **Werkstatt → Partner** tab manages the directory, and every task view can filter to partner-assigned work.
- **Werkstatt** (workshop / inventory) has shipped as a new top-level module with tabs for Dashboard, Bestand, Auf Baustelle, Nachbestellen, Projekt-Bedarfe, Katalog, Lieferanten, Partner, Kategorien & Lagerorte, Bestellungen, and Datanorm-Import. This release ships the schema and Partner tab wired to real endpoints; the remaining tabs render empty states until their backend endpoints land.
- **Project Gantt view** lets users see scheduled tasks for a project on a calendar timeline, with auto-scroll to today, weekend shading, and per-row actions.
- A **Construction / Office workspace switcher** in the sidebar lets users flip between the two mental models without losing context.
- The **task modal** learned about partners (`PartnerMultiSelect`) and the **project modal** learned about customers (`CustomerCombobox` with inline create and a "Vom Kunden · Stammdaten" snapshot).
- The initial page banner that said *"Internal Server Error"* on fresh deployments is fixed — it was caused by migrations not being applied; the new migrations are additive and preserve all existing data.

## Detailed changes

### 1. Customer management (Kunden)

- New `customers` table: `id`, `name`, `address`, `contact_person`, `email`, `phone`, `tax_id`, `notes`, `archived_at`, and audit fields. Projects gain a nullable `customer_id` FK (`SET NULL` on delete).
- Migration `20260501_0048_customers.py`:
  - Creates the `customers` table and adds `projects.customer_id`.
  - **Backfills** existing projects: groups them by normalised `(customer_name, customer_address)` tuple using the same `_normalize_key` rule already used by the CSV import pipeline. One `Customer` row per normalised group; projects link to it.
  - Legacy denormalised columns (`projects.customer_name`, `customer_address`, `customer_contact`, `customer_email`, `customer_phone`) are preserved as a **mirrored cache** — downstream consumers (construction-report PDF, xlsx export, weather-address resolver) keep working unchanged.
- New CRUD endpoints under `/api/customers` with search, aggregated project counts, and soft-archive. Permission: piggybacks on existing `projects:manage`.
- The project create/update endpoints now accept `customer_id`. When set, the five legacy customer fields are mirrored from the `Customer` row on every save. When only `customer_name` is sent (legacy API path or CSV import), the server does **match-or-create** using the same dedupe rule as the migration.
- New frontend surfaces:
  - `pages/CustomersPage.tsx` — searchable list with segmented `Alle · Aktiv · Archiviert` filter and table columns Name / Adresse / Kontakt / Projekte / Letzte Aktivität.
  - `pages/CustomerDetailPage.tsx` — two-column detail (Kontaktdaten / Notizen / Dateien on the left, Projekte with tabs and Aktivitäts-Feed on the right), plus a "+ Neues Projekt für diesen Kunden" shortcut.
  - `components/modals/CustomerModal.tsx` — create/edit with an `onSaved` callback used by the project modal to auto-select a freshly created customer.
  - `components/customers/CustomerCombobox.tsx` — keyboard-navigable combobox (↑/↓/Enter/Esc) with inline create.
- Sidebar gains a **Kunden** nav entry below Projects.

### 2. External partners (Subunternehmer / Fremdfirmen)

- New `partners` table and `task_partners` many-to-many join (mirrors the existing `task_assignments` shape — same columns, same sync semantics). A task can have 0..N partners and 0..N internal assignees independently.
- Migration `20260505_0049_partners.py` creates both tables (no backfill — partners are a new concept).
- New CRUD endpoints under `/api/partners` with search, `trade` filter, and soft-archive. Linked tasks accessible at `GET /api/partners/{id}/tasks`.
- Task create / update endpoints accept `partner_ids: number[]`. The server normalises (dedupe, drop archived), then syncs by deleting stale rows and inserting new ones. The `uq_task_partner(task_id, partner_id)` unique constraint enforces idempotent sync.
- Task list endpoints gain new query filters: `has_partners: bool` and `partner_id: int`.
- Task list responses now embed `partner_ids` and a lightweight `partners` array so the UI can render trade pills without an extra fetch.
- New frontend surfaces:
  - `pages/werkstatt/WerkstattPartnersTab.tsx` — partner directory, reachable from **Werkstatt → Partner**.
  - `pages/werkstatt/PartnerDetailOverlay.tsx` — right-side slide-over showing contact card, notes, and linked tasks with `Offen / Abgeschlossen` tabs.
  - `components/modals/PartnerModal.tsx` — create/edit with a `trade` typeahead that offers previously-used values to prevent "Elektro" / "Elektrik" divergence.
  - `components/partners/PartnerTradePill.tsx` — color-coded trade badge (Elektro amber, Sanitär blue, Maler purple, Dach orange, Fliesen teal, Gartenbau green, and a neutral default).
  - `components/partners/PartnerMultiSelect.tsx` — the multi-select field used inside `TaskModal` and `TaskEditModal`. Search + chip list + "+ Neuen Partner anlegen" inline create.
  - `components/partners/PartnerTaskChip.tsx` — compact chip shown on task rows (`MyTasks`, `OfficeTasks`, `ProjectTasks`) with overflow handling for multiple partners.
- A new **Nur Partner-Aufgaben** filter chip on MyTasksPage, OfficeTasksPage, and the project Tasks tab.

### 3. Werkstatt (Workshop / Inventory)

- New module with its own sidebar entry and top tab bar. Tabs: Dashboard · Bestand · Auf Baustelle · Nachbestellen · Projekt-Bedarfe · Katalog · Lieferanten · Partner · Kategorien & Lagerorte · Bestellungen · Datanorm-Import.
- Migration `20260425_0047_werkstatt_core.py` creates the core inventory schema.
- Mobile surfaces included: QR scanner, mobile home, mobile Artikel-Detail, mobile Nachbestellen list.
- The **Partner tab** is fully wired to real endpoints in this release. Other tabs render empty states until their backend endpoints land (see "Known limitations" below).
- The Werkstatt article, storage-location, category, and supplier models are shipped in the schema so backend endpoints can land incrementally without further migrations.

### 4. Project Gantt view

- New **Gantt** tab on the project detail page. Renders scheduled tasks on a horizontal timeline.
- Auto-scrolls to today on mount; today column is highlighted; weekend columns are subtly shaded.
- Each bar is clickable (opens task edit), with hover-reveal action row for Calendar export, mark-done, and Construction-Report shortcut when the task is owned by the current user.
- Unscheduled tasks render in a separate section below the timeline.
- Duration utility `formatDurationLabel` renders minutes as `"2h 30m"` / `"3h"` / `"45m"`.

### 5. Workspace switcher (Construction / Office)

- New top-of-sidebar segmented control lets users flip between **Construction** (Baustelle) and **Office** (Büro) workspace modes.
- Workspace-mode preference is persisted in `localStorage` and controls which sidebar entries and task surfaces are visible.

### 6. Task modal, project modal, and sidebar upgrades

- **Project modal**: the block of five raw customer inputs is replaced by a single customer combobox + a read-only `Vom Kunden · Stammdaten` snapshot + a `Abweichende Projekt-/Baustellenadresse verwenden` toggle that reveals a separate site-address textarea when construction happens somewhere other than the customer's billing address.
- **Task modal and task edit modal**: a new `Partner / Externe Firma` section carrying a `PartnerMultiSelect` alongside the existing assignee chip list.
- **Sidebar**: navy redesign with workspace switcher and cleaner separation between daily views (Overview / My Tasks / Calendar / Planning) and record views (Projects / Customers / Materials / Werkstatt / Wiki).
- **Header**: opens a more consistent breadcrumb / back-button flow for customer and partner detail views.

### 7. Construction report improvements

- The PDF / Excel report now accepts an explicit `company_name` argument and the submitted-by line uses the user's nickname when configured.
- Report-jobs infrastructure handles the company-name argument consistently across processing paths.

### 8. Tooling, hygiene, and release workflow

- `.gitignore` broadened to cover all dev / smoke / test SQLite databases (`apps/api/smoke*.db*`, `apps/api/*.db`) and the Claude Code scratch directory (`.claude/`) so local state never ends up in a release archive.
- The `mockData.ts` helper in `components/werkstatt/` is now empty — every `MOCK_*` constant is an empty array/string/zero. The file remains only to keep its pure utility functions and its type definitions; those are not placeholder content.
- `test_construction_report_uses_nickname_for_submitted_by` was failing on `main` prior to this release because the fake `build_report_pdf_bytes` fixture did not accept a `company_name` kwarg. The fixture is updated; the suite is now **180/180 green**.

## Upgrade notes

- Required migrations (applied additively — no existing rows are modified beyond adding a nullable `customer_id` to the `projects` table):
  - `20260425_0047_werkstatt_core.py`
  - `20260501_0048_customers.py`
  - `20260505_0049_partners.py`
- Apply with:
  - `docker compose run --rm api sh -lc 'cd /app && alembic upgrade head'`
- No environment variables or secrets need to change.
- **Data preservation**: the customer migration backfills existing projects by normalised `(customer_name, customer_address)` into `customers` rows. Legacy columns on `projects` are kept as a mirrored cache. No rows are deleted by the migration.
- **Permission layering**: all new customer/partner endpoints piggyback on the existing `projects:manage` permission. No new permission bits are introduced.
- **Mobile navigation** does NOT get a new bottom-tab slot for Kunden or Partner — both are accessible via the sidebar drawer on mobile and via drill-down from project / task rows.

## Known limitations

- The Werkstatt tabs other than **Partner** (Dashboard, Bestand, Auf Baustelle, Nachbestellen, Projekt-Bedarfe, Katalog, Lieferanten, Kategorien, Bestellungen, Datanorm-Import) render empty states because their dedicated `/api/werkstatt/*` read endpoints ship in a follow-up release. The schema is already in place, so those endpoints can land without further migrations.
- Deep-linking to a specific customer or partner via URL parameters is not yet supported; selection state lives in `AppContext`.
- The Gantt tab does not yet render partner chips on bars; partner association is currently only surfaced in list views.

## Summary

This release turns customers and partners from free-text fields on projects and tasks into first-class entities backed by the database. The biggest UX win is in the project modal, where picking an existing customer from a combobox replaces re-typing the same Stammdaten for every new job. The biggest structural win is the Werkstatt module, which arrives with its full schema and a working Partner tab; the remaining tabs are wired up visually but intentionally empty until their endpoints land.
