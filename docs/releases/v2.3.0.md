# v2.3.0 - Admin Backups page, Werkstatt suppliers UI, holiday pills

Date: 2026-04-28

## Highlights

- **New /admin/backups tab.** Operators can now list, create, download,
  upload, restore, and delete encrypted full-stack backups
  (`backup-<ts>.tar.enc`) entirely from the admin UI — no more shelling into
  the host. Mirrors the update-installer pattern: api delegates to the
  `update_runner` sidecar, which owns docker-socket access and runs the
  existing `scripts/backup.sh` / `scripts/restore.sh`.
- **Three new permissions: `backups:export`, `backups:restore`,
  `backups:manage`.** Splitting the existing single permission means a
  junior admin can download backups for off-site storage without holding
  the destructive-restore button. Admin role gets all three by default;
  non-admin roles get none until explicitly granted.
- **`BACKUP_PASSPHRASE` first-class setting.** Surfaced in `Settings` and
  `.env.example`, forwarded to the runner via docker-compose. The admin UI
  warns when the passphrase is empty so an operator doesn't trigger a
  backup that's silently going to fail.
- **Job-progress UX matches the existing system updater.** Restore shows
  a maintenance-mode banner and tail-of-log output, polled every 2 s
  until the runner reports a terminal status.
- **Type-to-confirm modal before restore.** Restore wipes the database via
  `pg_restore --clean` and replaces the uploads volume — friction
  proportional to consequence.
- **Werkstatt → Lieferanten is now a real CRUD page** instead of mock
  data. Backed by the existing `/api/werkstatt/suppliers` endpoints with
  list / create / edit / archive flows; the page now has a search +
  archived-filter, an inline create/edit form, and per-row archive
  buttons gated on `werkstatt:manage`.
- **Public holidays now appear on the personal time-tracking calendar**
  as an additional pill alongside vacation / school / sick leave. The
  separate "Feiertage (NRW)" callout below the calendar still
  summarises them as a list — the cell pill is purely additive so a day
  with both a holiday and an absence shows both signals.

## Detailed changes

### 1. Permissions split (`feat(rbac)`)

`apps/api/app/core/permissions.py` adds `backups:restore` and
`backups:manage` alongside the existing `backups:export`. They live under
the `system` permission group so they show up next to `system:manage` in
the role-permission matrix. The hard-coded `ROLE_ADMIN` set picks all
three up; other roles are unchanged.

### 2. `BACKUP_PASSPHRASE` setting (`feat(config)`)

`apps/api/app/core/config.py` gains `backup_passphrase: str = ""`. The api
itself never invokes the backup scripts — the variable is forwarded to the
`update_runner` container via `docker-compose.yml` (already in place since
v2.0.0) and surfaced in the admin UI through a `passphrase_configured`
boolean on the list response so operators can spot a missing value before
hitting the button.

`apps/api/.env.example` documents both `BACKUP_PASSPHRASE` and the
alternative `BACKUP_PASSPHRASE_FILE` mount.

### 3. update_runner: file-management module (`feat(runner)`)

New `apps/update_runner/app/backups.py` owns every filesystem operation
inside the runner's `/repo/backups` bind-mount:

- `is_valid_filename` and `safe_resolve` enforce a strict whitelist plus a
  defense-in-depth path-containment check (resolved Path must be
  `relative_to(BACKUP_DIR)`).
- `list_backups` returns metadata (size, mtime, generated-vs-uploaded
  flag) for every recognised archive.
- `write_uploaded_chunks` streams an upload via a `.tmp` sidecar and
  atomically renames into place — a crashed upload never leaves a
  half-written file masquerading as a real backup.
- `disk_usage` exposes free/total bytes so the UI can warn before the
  partition fills.

### 4. update_runner: generalised job model (`refactor(runner)`)

`apps/update_runner/app/jobs.py` was previously hard-coded for
`safe_update.sh`. It now models `Job.kind` as
`"update" | "backup" | "restore"`, sharing the same lock, log-path and
terminal-status plumbing across all three. The single-active-job
invariant survives intact and now also prevents a backup running during
an update rebuild — both touch the same docker stack.

New entry points: `queue_backup_job()` and
`queue_restore_job(filename=...)`.

### 5. update_runner: HTTP endpoints (`feat(runner)`)

`apps/update_runner/app/main.py` adds:

- `POST   /jobs/backup`             — kick off `scripts/backup.sh`.
- `POST   /jobs/restore`            — kick off `scripts/restore.sh <file>`.
- `GET    /backups`                 — list files + disk usage.
- `GET    /backups/{filename}`      — stream a backup (StreamingResponse;
  multi-GB downloads work without buffering).
- `POST   /backups/upload`          — accept a multipart upload, atomic
  rename into `backups/`.
- `DELETE /backups/{filename}`      — remove a file.

All write endpoints require the existing `X-Update-Token` header. The
existing `GET /jobs/{job_id}` doubles as the polling endpoint for
backup/restore jobs (same `Job` shape).

### 6. api: `update_runner_client` extensions (`feat(api)`)

`apps/api/app/services/update_runner_client.py` gains
`list_backups`, `queue_backup_job`, `queue_restore_job`, `delete_backup`,
`stream_backup_download`, `upload_backup`. Streaming endpoints use a
`Timeout(read=None)` so multi-GB transfers don't hit a spurious timeout
during `pg_dump`.

### 7. api: admin backup endpoints (`feat(api)`)

`apps/api/app/routers/admin.py` adds the operator-facing surface — every
call goes through `update_runner_client`, gated by the new permissions:

- `GET    /api/admin/backups`                       — `backups:manage`
- `POST   /api/admin/backups/full`                  — `backups:manage`
- `GET    /api/admin/backups/jobs/{job_id}`         — `backups:manage`
- `GET    /api/admin/backups/{filename}`            — `backups:export`
- `POST   /api/admin/backups/upload`                — `backups:restore`
- `POST   /api/admin/backups/{filename}/restore`    — `backups:restore`
- `DELETE /api/admin/backups/{filename}`            — `backups:manage`

Every state-changing call is audit-logged via the existing
`log_admin_action` helper (`backup.full.start`, `backup.full.upload`,
`backup.full.restore.start`, `backup.full.delete`,
`backup.full.download`).

Runner errors map to api status codes:
- `UpdateRunnerUnreachable` → 503
- `UpdateRunnerJobConflict` → 409 (with `active_job_id` in detail)
- 4xx from runner → forwarded verbatim
- anything else → 502

### 8. Frontend: AdminBackupsPanel (`feat(web)`)

New `apps/web/src/components/admin/AdminBackupsPanel.tsx`. Self-contained
component (kept out of the already-large `AdminPage` and `App.tsx`) that
renders:

- A table of existing backups with size, timestamp, source badge
  (Generated vs Imported).
- "Create full backup" button — kicks off the job and polls progress.
- Per-row Download / Restore / Delete buttons, gated on permission.
- An upload form (gated on `backups:restore`).
- `RestoreConfirmModal` that requires the operator to type the filename
  before the restore call goes through.
- A `BackupJobBanner` mirroring `AdminUpdateMenu`'s progress card; during
  a restore it adds an explicit "maintenance mode" warning since the
  stack restarts.

### 9. Frontend: state + polling (`feat(web)`)

`apps/web/src/App.tsx` gains `backupsList`, `backupJobProgress` and
`backupJobRunning` state plus six action functions
(`loadBackupsList`, `startFullBackup`, `startRestoreFromBackup`,
`downloadBackup`, `uploadBackup`, `deleteBackup`). The backup-job
polling loop mirrors `pollUpdateProgress` exactly: 2 s tick, 4 s on
transient error, terminal status flips the running flag and refreshes
the list.

`AdminPage.tsx` adds a `"backups"` tab visible when the user holds
`backups:manage` or `backups:restore`.

### 10. Tests

Three new test files:

- `apps/api/tests/test_backups.py` — permission gates, happy-path proxy,
  error translation (404/409/503), audit logging.
- `apps/update_runner/tests/test_backups.py` — exhaustive path-traversal
  rejection, write/read/delete round-trip, listing skips unrecognised
  files.
- `apps/update_runner/tests/test_main.py` — HTTP-level: upload→list,
  download streams full body, restore returns 404 on missing,
  `subprocess.run` stub verifies `backup.sh` job lifecycle.

The conftest fixture `_isolate_update_runner_client` is extended to
neutralise the new helpers by default — opt-in stubs in individual tests.

### 11. Frontend: Werkstatt suppliers CRUD (`feat(werkstatt)`)

`apps/web/src/pages/werkstatt/WerkstattLieferantenPage.tsx` was
previously a placeholder rendering `MOCK_SUPPLIERS`. It now talks to the
real `/api/werkstatt/suppliers` endpoints (list / create / patch /
archive) via a new `apps/web/src/utils/werkstattSuppliersApi.ts`
client. Features:

- Search-as-you-type filter against name + contact + city.
- "Archivierte anzeigen" toggle to surface soft-archived rows.
- Inline create + edit form with the full supplier schema (name, short
  name, email, order email, phone, contact person, address, lead-time
  days, notes).
- Per-row archive / unarchive buttons, gated client-side on
  `werkstatt:manage` (server re-checks the same gate).
- Empty-state messages and loading shimmer that match the existing
  Werkstatt visual language.

The `WerkstattSupplier` and `WerkstattSupplierCreate` types in
`apps/web/src/types/werkstatt.ts` are now consumed; the older duplicated
type in `utils/datanormApi.ts` is intentionally untouched (it covers a
narrower subset for the Datanorm-import dropdown).

### 12. Public holidays in the personal time-tracking calendar (`feat(time)`)

`apps/web/src/pages/TimePage.tsx`'s `monthCalendar` memo now merges
`publicHolidays` into the day cells alongside the v2.2.0 vacation /
school-absence pills. Holidays are NOT scoped by user — they apply to
everyone in the configured region (NRW today) — so the loop intentionally
skips the `targetUserId` filter the absence loops use. The pill picks up
the existing `.time-calendar-cell--absence-holiday` styling so days
with both an absence and a holiday show both signals stacked.

The `monthHolidays` callout below the calendar (the list with the
"Feiertage (NRW)" header) is unchanged — the pill is purely additive.

## Migrations

None. This release is API + UI only; no DB schema changes.

## Compatibility / breaking changes

- Existing `POST /api/admin/backups/database` (DB-only export with a
  per-export key file) is **unchanged** and still gated by
  `backups:export`. The new full-archive flow is purely additive.
- Roles other than admin do not pick up `backups:restore` or
  `backups:manage` automatically. Operators who relied on giving the
  `accountant` or `planning` role `backups:export` for off-site copies
  see no behavior change; if they want those roles to manage the
  full-archive listing they must grant the new permission explicitly via
  the role-permissions UI.

## Rollout

Standard `safe_update.sh --pull --branch main`. After the upgrade:

1. Set `BACKUP_PASSPHRASE` in your production env (long random value;
   `openssl rand -base64 48` is fine). Store it off-site — losing it
   means existing backups are unrecoverable.
2. Restart the stack (the safe_update flow already does this) so the
   `update_runner` container picks up the new env var.
3. Visit `/admin/backups` and trigger a backup to verify the round-trip.
   The list refreshes when the job completes.

If the runner sidecar isn't running on a stack (legacy deployment),
`/admin/backups` returns 503 and the UI surfaces a clear "runner not
reachable" notice — the legacy DB-only export under the System tab
continues to work either way.
